# Linux Systems — Lesson Plan

Linux is the cross-cutting substrate of this curriculum. It is the operating system that runs on embedded targets (topic 04), terminates and shapes network traffic (topic 06), hosts the signal-processing and radar tooling you'll build (topic 08), and serves the AI/ML stacks you'll deploy (topic 09). Almost every modern engineering job — embedded, networking, DevOps, ML infrastructure, defense systems — assumes fluency with a Linux command line and a working mental model of the kernel/userspace boundary. This module exists to turn the Linux you've *used* over a software career into Linux you *understand* from the boot vector up.

You arrive with strong prerequisites: a software developer's comfort with files, processes, and code; an EE/radar background that makes hardware, registers, interrupts, and device-level reasoning feel natural (this pays off enormously in the device-driver, device-tree, and embedded-Linux modules). Because of that, we move briskly through shell and coreutils basics, then go deep and thorough where the real leverage is: systemd, the kernel/userspace contract, system calls, /proc and /sys, drivers, storage, containers, and a full embedded-Linux build-and-boot module. As requested, nothing is skipped for being "too easy" — the early modules are compact refreshers with drills, not omissions. MODERN additions (containers, eBPF, infrastructure-as-code) are flagged inline so you can see exactly where today's market has moved past the Linux of a decade ago.

## Learning outcomes

By the end of this module you will be able to:

- Navigate, manipulate, and script the Linux environment fluently from a shell, including text-processing pipelines with grep/sed/awk.
- Explain the filesystem hierarchy, the permission/ownership model (including setuid, sticky, ACLs, capabilities), and reason about security implications.
- Describe and inspect the full boot chain from firmware (BIOS/UEFI) through bootloader, kernel, and init, and configure systemd units, services, timers, and journald.
- Articulate the kernel/userspace boundary, trace system calls, and read system state through /proc and /sys.
- Understand device files, udev, kernel modules, and write a trivial loadable kernel module and character-device driver.
- Manage storage: partitions, filesystems, LVM, and basic RAID; reason about journaling and mount options.
- Configure and inspect Linux networking with the modern `ip`/`ss`/`nft` tooling and hand off cleanly to topic 06.
- Write robust bash scripts and know when to reach for Python instead for operations work.
- Build software natively and cross-compile for another architecture.
- Build, run, and reason about containers (Docker/Podman) and explain the namespaces/cgroups primitives beneath them. (MODERN)
- Build a custom embedded Linux image with Yocto or Buildroot, understand device trees and BSPs, and boot it on a single-board computer.
- Explain what eBPF is and why it has reshaped observability, networking, and security. (MODERN)

## Module breakdown

### Module 1: Shell fundamentals & coreutils
- **Concepts:** What a shell is (bash vs zsh vs dash/POSIX sh, login vs interactive vs non-interactive); the read-eval loop; command vs builtin vs function vs alias. Anatomy of a command line: words, quoting (single vs double vs none, `$'...'`), escaping, globbing/wildcards (`*`, `?`, `[]`, `**` globstar), brace expansion, tilde expansion. Variables and environment (`export`, `env`, `set`, `PATH`, `$?`, `$$`, `$!`, `$#`, `$@` vs `$*`, positional params). Command substitution `$(...)`, arithmetic `$((...))`. Redirection (`>`, `>>`, `<`, `2>`, `&>`, `2>&1`, here-docs `<<`, here-strings `<<<`, `/dev/null`, fd duplication). Pipes and the pipeline model; `tee`. Exit codes, `&&`/`||`/`;`. History and expansion (`!!`, `!$`, Ctrl-R). Tab completion. Core coreutils survey: `ls`, `cp`, `mv`, `rm`, `mkdir`, `cat`, `head`, `tail`, `less`, `wc`, `sort`, `uniq`, `cut`, `tr`, `find`, `xargs`, `stat`, `file`, `ln` (hard vs symbolic), `du`, `df`, `which`/`type`/`command`, `tee`, `echo` vs `printf`, `seq`, `date`. Getting help: `man` (sections 1-8), `info`, `--help`, `apropos`, `tldr` (MODERN).
- **Why it matters / connections:** Everything else in this module runs through a shell. Quoting and redirection bugs are the single most common source of broken scripts. Underpins the scripting module and every hands-on exercise in topics 06/08/09.
- **Hands-on / exercises:** CLI warm-up drill: find the 10 largest files under `/var`, sort processes by memory, count unique IP prefixes in a log. Build a one-liner pipeline using `find ... -print0 | xargs -0`. Demonstrate the difference between `$@` and `$*` and between hard and symbolic links with `stat`. Reproduce a quoting bug (filename with spaces) then fix it.
- **You've got it when:** You can construct a multi-stage pipeline without trial-and-error, predict what globbing/quoting will expand to before pressing Enter, and explain why `for f in $(ls)` is a bug.

### Module 2: Filesystem hierarchy & permissions
- **Concepts:** Filesystem Hierarchy Standard (FHS): `/`, `/bin`, `/sbin`, `/usr` (and the `/usr` merge — MODERN), `/etc`, `/var`, `/tmp` vs `/var/tmp`, `/home`, `/root`, `/opt`, `/srv`, `/boot`, `/dev`, `/proc`, `/sys`, `/run`, `/lib`/`/lib64`, `/mnt`, `/media`. Everything-is-a-file philosophy; inodes, hard links, the dentry/path distinction. Permission model: user/group/other, read/write/execute and what x means on a directory; octal vs symbolic (`chmod 755` vs `chmod u+x`); `chown`/`chgrp`; umask. Special bits: setuid, setgid (and setgid on directories for group inheritance), the sticky bit (`/tmp`). POSIX ACLs (`getfacl`/`setfacl`). Linux capabilities (`getcap`/`setcap`, e.g. `cap_net_raw`) as the modern alternative to blanket setuid root (MODERN). Extended attributes (`xattr`), immutability (`chattr +i`). Symlink vs hardlink semantics across filesystems.
- **Why it matters / connections:** Permission and capability reasoning is core to security and to embedded systems where you run with least privilege. The FHS map tells you where to look when something breaks. Connects to containers (filesystem layering) and to the device-files module (`/dev`, `/proc`, `/sys`).
- **Hands-on / exercises:** Walk the FHS on a live system and explain each top-level dir in one sentence. Create a shared group directory using setgid so all members can collaborate. Reproduce the classic setuid-root scenario with `ping`/`passwd`, then show the capabilities-based equivalent. Use ACLs to grant one extra user access without changing the group.
- **You've got it when:** Given a permission string like `drwxr-sr-t` you can decode every bit, and you can explain why `cap_net_raw` is safer than setuid root.

### Module 3: Processes, signals & job control
- **Concepts:** Process model: PID, PPID, process groups, sessions, the process tree (`pstree`), init as PID 1, orphans and reparenting, zombies and reaping (`wait`). Process states (R/S/D/Z/T) and what uninterruptible sleep (D) means. `fork`/`exec`/`exit`/`wait` lifecycle. Viewing: `ps` (BSD vs SysV syntax), `top`/`htop`, `pgrep`/`pkill`. Signals: the standard set (`SIGTERM`, `SIGKILL`, `SIGINT`, `SIGHUP`, `SIGSTOP`/`SIGCONT`, `SIGCHLD`, `SIGSEGV`, `SIGUSR1/2`), what's catchable vs uncatchable (KILL/STOP), default dispositions, `kill`/`killall`, `trap` in scripts. Job control: foreground/background (`&`), `jobs`, `fg`, `bg`, `Ctrl-Z`, `disown`, `nohup`, `setsid`. Priorities: `nice`/`renice`, scheduling classes (intro). Resource limits: `ulimit`, `getrlimit`. Environment inheritance across exec.
- **Why it matters / connections:** Signals and process lifecycle are exactly how systemd supervises services (Module 6) and how containers manage PID 1 (Module 14). The fork/exec model is the foundation under the system-calls module. Job control matters daily at the terminal.
- **Hands-on / exercises:** Create a zombie deliberately and observe it in `ps`, then reap it. Write a script that installs a `trap` for `SIGTERM` to clean up on exit. Background a long job, detach it with `disown`, confirm it survives logout. Use `nice`/`renice` and observe CPU share with `top`.
- **You've got it when:** You can explain why `SIGKILL` can't be trapped, why a process in D-state won't die, and what PID 1 must do that ordinary processes need not.

### Module 4: Users, groups & authentication
- **Concepts:** UID/GID model, `/etc/passwd`, `/etc/shadow` (and password hashing formats), `/etc/group`, `/etc/gshadow`. System vs regular users, the root account, service accounts and `nologin` shells. User management: `useradd`/`usermod`/`userdel`, `groupadd`, `passwd`, `id`, `groups`, `who`/`w`/`last`. Privilege escalation: `su` vs `sudo`, `/etc/sudoers` and `visudo`, `sudo -e`, NOPASSWD, the difference between `sudo -i` and `sudo command`. PAM (Pluggable Authentication Modules) at an architectural level — the stack, modules, common configs. Login flow and the difference between a login shell's environment and a service's. SSH key-based auth as the practical day-to-day (cross-ref topic 06). Brief note on nsswitch.conf and directory-backed auth (LDAP/SSSD) in enterprise contexts.
- **Why it matters / connections:** Least-privilege design, service accounts, and sudo policy are job-relevant security topics. PAM and nsswitch explain "how does login actually work." Connects to systemd service hardening (Module 6) and SSH in networking (topic 06).
- **Hands-on / exercises:** Create a service account with a `nologin` shell. Add yourself to a group and prove you must re-login for it to take effect. Configure a sudoers rule allowing one specific command without a password (via `visudo`). Inspect a password hash field in `/etc/shadow` and identify the algorithm.
- **You've got it when:** You can explain the journey from typing a password to a running shell, and why group membership changes need a new session.

### Module 5: The boot chain — firmware to init
- **Concepts:** Power-on to userspace, in order. Firmware: BIOS (legacy, MBR, 512-byte boot sector) vs UEFI (GPT, EFI System Partition, `.efi` applications, Secure Boot, `efibootmgr`) — MODERN emphasis on UEFI. Bootloader: GRUB2 (config, `grub.cfg`, stages, `grub-mkconfig`), and embedded alternatives U-Boot and systemd-boot. What the bootloader loads: the kernel image (`vmlinuz`, bzImage), the initramfs/initrd (why it exists — drivers to mount the real root), and the kernel command line. Kernel hand-off: decompression, hardware discovery, mounting initramfs, then pivoting to the real root filesystem. The init handoff: kernel execs PID 1. History: SysV init and runlevels (for context) vs modern systemd targets. Kernel command-line parameters (`root=`, `ro`, `quiet`, `console=`, `init=`). Boot diagnostics: `dmesg`, `journalctl -b`, `systemd-analyze`.
- **Why it matters / connections:** This is the spine that ties firmware (your hardware comfort zone) to userspace. It's directly load-bearing for the embedded-Linux module (U-Boot, custom kernel cmdline, booting on an SBC) and for debugging machines that won't boot. The initramfs concept recurs in containers and recovery.
- **Hands-on / exercises:** Read your machine's kernel command line from `/proc/cmdline` and explain each token. Inspect the EFI boot entries with `efibootmgr`. Extract and list the contents of an initramfs (`lsinitramfs`/`unmkinitramfs`). Run `systemd-analyze blame` and `systemd-analyze critical-chain` to see what dominates boot time.
- **You've got it when:** You can narrate, without notes, every stage from pressing the power button to the login prompt, and explain why an initramfs is needed at all.

### Module 6: systemd — units, services, journald (MODERN, deep)
- **Concepts:** Why systemd replaced SysV init (parallelism, dependency tracking, socket/dbus activation, cgroup-based supervision, unified logging). Unit concept and unit types: `.service`, `.socket`, `.target`, `.timer`, `.mount`, `.automount`, `.path`, `.device`, `.slice`, `.scope`. Unit file anatomy: `[Unit]` (Description, After, Requires, Wants, Before, Conflicts), `[Service]` (ExecStart/Stop/Reload, Type=simple/forking/oneshot/notify/exec, Restart=, User=, WorkingDirectory=, Environment=), `[Install]` (WantedBy). Where units live: `/usr/lib/systemd/system` vs `/etc/systemd/system` (and override precedence, drop-ins via `systemctl edit`). `systemctl` verbs: start/stop/restart/reload, enable/disable, status, is-active, mask/unmask, daemon-reload, list-units, list-unit-files. Targets vs runlevels (multi-user.target, graphical.target, rescue, emergency). Timers as the cron replacement (OnCalendar, OnBootSec, persistent timers) — MODERN. Socket activation. Service hardening directives (ProtectSystem, PrivateTmp, NoNewPrivileges, CapabilityBoundingSet, SystemCallFilter) — MODERN security. journald: structured/binary logs, `journalctl` (`-u`, `-b`, `-f`, `-p`, `--since`, `-k`), persistent vs volatile storage, rate limiting, forwarding to syslog. User services (`systemctl --user`) and `logind`. `systemd-resolved`, `systemd-networkd` (overview; cross-ref topic 06).
- **Why it matters / connections:** systemd is *the* init and service manager on essentially every modern non-embedded Linux and many embedded ones. Writing a correct unit file is a core practical skill. Builds directly on processes/signals (Module 3) and feeds the capstone, containers, and embedded modules.
- **Hands-on / exercises:** Write a `.service` unit for a small program you wrote (e.g., a Python or bash daemon), with `Restart=on-failure` and a dedicated `User=`. Enable it, kill it, and watch it restart. Add a `.timer` that runs a backup script every night and verify with `systemctl list-timers`. Use `systemctl edit` to add a drop-in override. Read its logs with `journalctl -u`. Harden it with `ProtectSystem=strict` and `NoNewPrivileges=yes`.
- **You've got it when:** You can author a working hardened service + timer from scratch, explain the difference between `Wants=` and `Requires=`, and find any service's logs instantly.

### Module 7: Kernel vs userspace & system calls
- **Concepts:** The privilege ring model (kernel mode vs user mode, ring 0 vs ring 3), the trap/syscall instruction as the controlled gateway. What the kernel provides: process scheduling, memory management (virtual memory, paging, MMU — ties to your hardware background), the VFS, networking stack, device drivers. The C library (glibc/musl) as the thin wrapper over raw syscalls. The syscall ABI vs API. Key syscall families: process (`fork`, `execve`, `wait4`, `exit`, `clone`), file (`open`, `read`, `write`, `close`, `lseek`, `stat`, `mmap`), IPC (`pipe`, `socket`, `shmget`, `futex`), signals (`kill`, `sigaction`, `rt_sigprocmask`). Tracing tools: `strace` (and how to read it), `ltrace`, `/proc/<pid>/syscall`. errno and how errors propagate. Context switches, interrupts (hardware/software), the difference between a syscall and an interrupt. User/kernel data copying (`copy_to_user`/`copy_from_user`) as a preview of the driver module.
- **Why it matters / connections:** This boundary is the conceptual core of "how Linux works." `strace` is one of the most powerful debugging tools you'll own. Directly precedes the device-driver module and underpins containers (namespaces are syscall-level). Your radar/embedded interrupt experience maps cleanly onto kernel interrupt handling.
- **Hands-on / exercises:** `strace -f` a simple program (e.g., `cat file`) and identify every syscall it makes and why. Write a minimal C program that calls `write(2)` directly with no stdio, and compare with `strace`. Use `strace -c` to get a syscall histogram of a real command. Find a failing syscall in a misbehaving program via `strace` and read the errno.
- **You've got it when:** You can explain what physically happens on a `read()` call from user code to disk and back, and you reach for `strace` instinctively when a program "just fails."

### Module 8: /proc, /sys and system introspection
- **Concepts:** procfs as a window into the kernel: `/proc/<pid>/` (cmdline, environ, fd, maps, status, stat, limits, cwd, root), `/proc/cpuinfo`, `/proc/meminfo`, `/proc/loadavg`, `/proc/mounts`, `/proc/cmdline`, `/proc/version`, `/proc/interrupts`, `/proc/sys/` (the sysctl tree). sysfs (`/sys`): the device/driver/bus model exposed as files — `/sys/class`, `/sys/block`, `/sys/devices`, `/sys/module`. `sysctl` for tuning kernel parameters (`net.ipv4.ip_forward`, `vm.swappiness`, etc.) and persistent config in `/etc/sysctl.d/`. The relationship between /proc, /sys, and the device model. Modern observability layer: `/proc`-based tools (`vmstat`, `iostat`, `mpstat`, `free`, `uptime`) and the BPF-based successors (`bpftop`, bcc/bpftrace) — MODERN, forward-reference to Module 16.
- **Why it matters / connections:** This is how you *observe* everything from earlier modules without special tools — processes, memory, the device tree. sysfs is the userspace face of the driver/udev model (Module 9). sysctl tuning is real production/networking work (cross-ref topic 06).
- **Hands-on / exercises:** Find a running process's open file descriptors via `/proc/<pid>/fd`. Read `/proc/meminfo` and reconcile it with `free -h`. Toggle `net.ipv4.ip_forward` via `sysctl` and make it persistent. Explore `/sys/class/net/` and map it to `ip link`. Read `/proc/interrupts` and identify which devices generate the most interrupts (ties to your radar/hardware instincts).
- **You've got it when:** You can answer "how much memory is this process using / what files does it have open / how do I turn on IP forwarding" using only `/proc` and `/sys`.

### Module 9: Device files, udev, modules & drivers (intro)
- **Concepts:** Device files in `/dev`: character vs block devices, major/minor numbers, `mknod` (historical), null/zero/random/full/tty/loop. The dynamic `/dev`: `devtmpfs` + udev. udev: the device manager — events from the kernel (uevents over netlink), rules in `/etc/udev/rules.d/` and `/lib/udev/rules.d/`, matching on attributes, naming/symlinking devices (e.g., predictable network interface names, persistent device naming), `udevadm info`/`monitor`/`trigger`. Kernel modules: what a `.ko` is, `lsmod`, `modinfo`, `insmod`/`rmmod`/`modprobe`, module dependencies (`depmod`, `modules.dep`), autoloading, module parameters, `/etc/modprobe.d/` and blacklisting, the kernel taint flag. Driver intro: the character-device driver model (file_operations, register_chrdev, the `cdev` interface), the platform-device and device-tree binding model (forward-ref to embedded module), how a driver maps hardware (MMIO, request_irq) — connecting directly to your radar/embedded experience.
- **Why it matters / connections:** This is where your hardware/EE background becomes a superpower. Drivers, udev rules, and modules are central to embedded Linux (Module 18) and to bringing up custom hardware. Builds on syscalls (Module 7) and /sys (Module 8).
- **Hands-on / exercises:** `lsmod` and `modinfo` a loaded module; load and unload one safely. Write a udev rule that creates a stable symlink for a USB serial device by its vendor/product ID (`udevadm info`). **Write a minimal "hello world" loadable kernel module** (module_init/module_exit, `printk`), build it against your kernel headers, `insmod` it, and read the message via `dmesg`. Extend it into a **trivial character-device driver** exposing `/dev/mychardev` with read/write `file_operations`.
- **You've got it when:** You've loaded your own kernel module, exposed a working `/dev` node from your own driver, and can write a udev rule that survives reboots.

### Module 10: Package management & software installation
- **Concepts:** The package-manager landscape by family. Debian/Ubuntu: `dpkg` (low level), APT (`apt`/`apt-get`/`apt-cache`), `.deb` anatomy, sources lists, `/etc/apt/`, holds, `apt-mark`. Red Hat/Fedora: RPM, `dnf`/`yum`, repos, `.rpm`. Arch: `pacman` (for breadth). Concepts that generalize: repositories and mirrors, package signing/GPG and trust, dependency resolution, version pinning, the difference between a package and what it installs (use `dpkg -L`/`rpm -ql`). Building from source the classic way (`./configure && make && make install`) and why packaging exists. MODERN: universal/sandboxed package formats — Flatpak, Snap, AppImage — and their trade-offs; language-level package managers (pip/venv, npm, cargo) and the system-vs-language-package tension; Nix as a reproducible alternative (brief). Keeping a system updated and the security-update workflow (`unattended-upgrades`).
- **Why it matters / connections:** You install everything else through these. Repo/signing concepts matter for security. The system-vs-language package tension is a daily DevOps reality and motivates containers (Module 14). Cross-compilation (Module 13) and Yocto (Module 18) build packages too.
- **Hands-on / exercises:** Find which package owns a given file (`dpkg -S`/`rpm -qf`) and list a package's files. Add a third-party repo with its GPG key and install from it. Pin/hold a package at a version. Install the same tool via the system package manager and via a language manager, and compare where files land. Inspect a `.deb` with `ar`/`dpkg-deb`.
- **You've got it when:** You can confidently install, pin, inspect, and remove software on at least Debian- and RHEL-family systems and explain the security model of package signing.

### Module 11: Text processing — grep, sed, awk & friends
- **Concepts:** Regular expressions thoroughly: BRE vs ERE vs PCRE, anchors, character classes, quantifiers, groups, backreferences, greedy vs lazy. `grep`/`egrep`/`fgrep`/`rg` (ripgrep, MODERN): `-i`, `-v`, `-r`, `-o`, `-E`, `-P`, `-A/-B/-C`, `--include`. `sed`: substitution (`s///g`, flags, addresses, ranges), delete/print/insert/append, in-place editing (`-i`), holding space (intro). `awk`: the pattern-action model, fields (`$1`, `$NF`, `FS`/`OFS`), records, `BEGIN`/`END`, built-in variables (`NR`, `NF`, `FNR`), arithmetic and string functions, associative arrays, computing sums/averages/histograms from columnar data. Supporting tools: `cut`, `paste`, `join`, `sort -k`, `uniq -c`, `comm`, `column`, `tr`, `jq` for JSON (MODERN), `xargs` for fan-out. Building real pipelines that combine these.
- **Why it matters / connections:** Log analysis, config munging, and data extraction are constant in ops, networking (parsing `ss`/`ip` output), and signal/data work (topic 08). awk in particular turns ad-hoc data wrangling into one-liners. Foundation for the scripting module.
- **Hands-on / exercises:** From a web/server log: extract all 4xx requests with grep, count requests per IP with awk associative arrays, and produce a top-10 with `sort | uniq -c | sort -rn | head`. Use `sed -i` to rewrite a config value across many files safely. Write an awk script computing the mean and max of a numeric column. Parse JSON output of a command with `jq`.
- **You've got it when:** You instinctively reach for awk to summarize columnar data, can write a non-trivial regex without a reference, and can transform structured text without leaving the shell.

### Module 12: Bash scripting (and a note on Python for ops)
- **Concepts:** Script structure: shebang (`#!/usr/bin/env bash`), `set -euo pipefail` and why it matters, `IFS`. Variables and quoting discipline, parameter expansion in depth (`${var:-default}`, `${var:?}`, `${var#pattern}`, `${var/old/new}`, `${#var}`, slicing). Arrays (indexed and associative). Control flow: `if`/`elif`, `[[ ]]` vs `[ ]` vs `test`, arithmetic `(( ))`, `case`, `for`/`while`/`until`, `select`. Functions, `local`, return values vs output, `readonly`. Argument parsing: `$@`, `shift`, `getopts`. I/O: reading files line-by-line (`while read -r`), here-docs, process substitution `<(...)`. Error handling: exit codes, `trap ... EXIT/ERR`, cleanup. Debugging: `bash -x`, `set -x`, ShellCheck (MODERN — static analysis you should always run). Writing portable POSIX sh vs bash-isms. **The note on Python for ops:** when a script grows past ~50-100 lines, needs data structures, error handling, HTTP/JSON, or testing, switch to Python; `subprocess`, `pathlib`, `argparse`, `os`/`shutil`, and `venv`; the boundary heuristic for choosing bash vs Python.
- **Why it matters / connections:** Glue scripts run the operational world — backups, deploys, CI steps, device provisioning. Robust scripting (with `set -euo pipefail` and ShellCheck) separates professionals from hackers. Feeds systemd services/timers (Module 6), containers (entrypoints), and CI in topic 10.
- **Hands-on / exercises:** Write a backup script that takes options via `getopts`, validates input, uses `trap` to clean up a temp dir, and logs to journald-friendly stderr. Run ShellCheck on it and fix every finding. Rewrite a gnarly 80-line bash script in Python and reflect on which is clearer. Write a script that safely iterates over filenames with spaces/newlines.
- **You've got it when:** Your scripts start with `set -euo pipefail`, pass ShellCheck clean, handle errors and cleanup, and you can articulate exactly when to abandon bash for Python.

### Module 13: Build systems & cross-compilation
- **Concepts:** The native build pipeline: preprocessor → compiler → assembler → linker; `gcc`/`clang`, object files, static vs dynamic libraries (`.a` vs `.so`), the dynamic linker/loader (`ld.so`, `ldd`, `LD_LIBRARY_PATH`, `rpath`, `ldconfig`). Build orchestration: `make` and Makefiles (targets, prerequisites, variables, pattern rules), then modern systems — CMake (MODERN, the de facto standard), Ninja, Meson; autotools for legacy context. Toolchains: what a toolchain is (compiler + binutils + libc + headers), the host/build/target triplet concept. **Cross-compilation:** why and when (building for ARM on an x86 host), cross-toolchains (`arm-linux-gnueabihf-gcc`, `aarch64-linux-gnu-gcc`), sysroots, `CC`/`CROSS_COMPILE`, the difference between cross-compiling, emulation (QEMU user-mode — MODERN), and native-on-target builds. Reproducible builds (intro). Linking pitfalls (ABI mismatch, glibc vs musl).
- **Why it matters / connections:** Cross-compilation is the gateway to embedded Linux (Module 18) and to your radar/embedded target work (topic 04). The linker/loader model explains a whole class of "library not found" runtime errors. CMake fluency is broadly expected in C/C++ jobs.
- **Hands-on / exercises:** Build a small C program with `gcc`, then build it as a shared library and link against it; inspect with `ldd` and `nm`. Write a minimal CMakeLists.txt and build with Ninja. **Cross-compile a "hello world" for ARM** (aarch64) on your x86 machine and run it under `qemu-aarch64`. Diagnose a deliberate `LD_LIBRARY_PATH`/rpath problem.
- **You've got it when:** You can cross-compile a binary for an ARM SBC and explain every component of the toolchain and the host/target triplet.

### Module 14: Containers — Docker/Podman & the primitives beneath (MODERN, deep)
- **Concepts:** What a container actually is: *not* a VM — shared kernel, isolated userspace. The Linux primitives: **namespaces** (PID, mount, network, UTS, IPC, user, cgroup, time) — each isolates one resource class; **cgroups v2** (MODERN) — resource limiting/accounting for CPU, memory, IO, PIDs; **capabilities** and **seccomp** and **overlayfs** (union/layered filesystems) as the storage mechanism. How `unshare`/`clone` create namespaces (ties back to syscalls, Module 7). Docker: images vs containers, layers and the build cache, Dockerfile instructions (FROM, RUN, COPY, ENV, ENTRYPOINT vs CMD, EXPOSE, WORKDIR, multi-stage builds), registries, tags/digests, volumes and bind mounts, container networking (bridge, host, none) and port mapping, `docker compose` for multi-container apps. **Podman** (MODERN): the daemonless, rootless, fork/exec model and Docker-CLI compatibility — increasingly preferred in security-conscious/enterprise/RHEL environments; rootless containers; generating systemd units from containers (Quadlet). Best practices: minimal base images (distroless/alpine), non-root users, image scanning, not baking secrets. OCI standards. Brief orchestration horizon: Kubernetes exists and is where this scales (pointer, not a deep dive).
- **Why it matters / connections:** Containers are the dominant modern deployment unit and a near-universal job-market expectation. They directly solve the system-vs-language package tension from Module 10. The namespace/cgroup primitives are pure applied kernel knowledge from Modules 3, 7, 8. Host the AI/ML stacks of topic 09 and reproducible build/test environments for topic 10.
- **Hands-on / exercises:** Write a multi-stage Dockerfile that builds a small app and ships a minimal final image running as a non-root user. Run it with Docker, then run the same image with Podman rootless. Use `docker compose` to stand up an app + database. **Demonstrate the primitives directly:** use `unshare -p -f --mount-proc` to get a new PID namespace and watch your shell become PID 1; inspect a container's cgroup limits under `/sys/fs/cgroup`. Generate a systemd unit from a Podman container.
- **You've got it when:** You can write a production-shaped Dockerfile, explain a container as "namespaces + cgroups + a layered filesystem over a shared kernel," and run the same image under both Docker and rootless Podman.

### Module 15: Storage, filesystems & LVM
- **Concepts:** The block-device stack from disk to mountpoint. Disks and partitioning: MBR vs GPT, `lsblk`, `fdisk`/`gdisk`/`parted`. Filesystems: ext4 (journaling — ordered/writeback/journal modes), XFS, Btrfs and ZFS (copy-on-write, snapshots, checksums — MODERN), FAT/exFAT for interop, tmpfs (RAM-backed), overlayfs (ties to containers). Creating/checking: `mkfs.*`, `fsck`, `tune2fs`. Mounting: `mount`/`umount`, `/etc/fstab` (fields, mount options like `noatime`, `ro`, `nosuid`), UUIDs/labels vs device names, systemd mount units, `findmnt`. **LVM** (Logical Volume Manager): physical volumes → volume groups → logical volumes, why it exists (online resize, snapshots, pooling), `pvcreate`/`vgcreate`/`lvcreate`/`lvextend`, thin provisioning. Swap (partition vs file, swappiness). RAID basics (`mdadm`, levels 0/1/5/10, and LVM/Btrfs/ZFS RAID). Encryption: LUKS/dm-crypt (intro). Monitoring: `df`, `du`, `iostat`, SMART (`smartctl`).
- **Why it matters / connections:** Provisioning storage, resizing volumes without downtime, and choosing filesystems are core sysadmin/DevOps skills. overlayfs and tmpfs tie directly to containers (Module 14). fstab/mount units connect to systemd (Module 6) and boot (Module 5). Embedded targets use specialized FS (squashfs, UBIFS) — forward-ref Module 18.
- **Hands-on / exercises:** Using loopback files as fake disks: create partitions, `mkfs.ext4`, mount via UUID in `/etc/fstab`. **Build an LVM stack** from two loop devices: PV → VG → LV, format and mount it, then `lvextend` and grow the filesystem online. Create and restore an LVM (or Btrfs) snapshot. Create a LUKS-encrypted volume and unlock it. Inspect everything with `lsblk -f`.
- **You've got it when:** You can grow a logical volume and its filesystem on a live system, explain journaling vs copy-on-write, and write a correct UUID-based fstab entry.

### Module 16: Networking from the Linux side (hand-off to topic 06)
- **Concepts:** This module covers the *Linux mechanisms*; topic 06 covers the protocols. The modern `iproute2` suite (MODERN — `ifconfig`/`route`/`netstat` are deprecated): `ip addr`, `ip link`, `ip route`, `ip neigh`, network namespaces (`ip netns` — ties to containers). Socket/connection inspection: `ss` (replacing netstat), `lsof -i`. Interface configuration: NetworkManager, systemd-networkd, netplan (Ubuntu) — the competing modern configuration layers. DNS resolution path: `/etc/resolv.conf`, `/etc/nsswitch.conf`, `/etc/hosts`, systemd-resolved. The kernel packet path and **firewalling: iptables vs the modern nftables** (MODERN — `nft`, tables/chains/rules, NAT, the netfilter framework beneath both); a note on the higher-level frontends (`ufw`, firewalld). Linux as a router (`ip_forward`), bridges (`ip link add type bridge`), VLANs, bonding/teaming, virtual interfaces (veth pairs — the container networking primitive), tunnels (intro). Diagnostics: `ping`, `traceroute`, `mtr`, `tcpdump` (capture/read), `ethtool`. Bandwidth/socket stats. **Clean hand-off to topic 06:** Linux gives you the *interfaces, sockets, routing table, and firewall*; topic 06 explains TCP/IP, routing protocols, and what's on the wire.
- **Why it matters / connections:** Every networked service, container, and embedded device touches this layer. nftables and iproute2 fluency is current and expected; many engineers still only know the deprecated tools. veth/netns are exactly how container networking (Module 14) works. Directly precedes and feeds topic 06.
- **Hands-on / exercises:** Inspect your machine with `ip addr`, `ip route`, and `ss -tulpn`; map each listening socket to its process. Create two network namespaces connected by a veth pair and ping across them (a from-scratch model of container networking). Write an `nft` ruleset that allows SSH and drops everything else inbound. Capture and read a handful of packets with `tcpdump`. Turn the machine into a router between two interfaces.
- **You've got it when:** You use `ip`/`ss`/`nft` reflexively instead of the legacy tools, can build a veth-connected namespace pair, and can write a basic stateful firewall.

### Module 17: eBPF & modern observability (MODERN, intro)
- **Concepts:** What eBPF is: a safe, verified, JIT-compiled virtual machine *inside the kernel* that runs sandboxed programs attached to hooks (kprobes/uprobes, tracepoints, network XDP/tc, LSM) without writing a kernel module or rebooting. Why it's transformative: dynamic, low-overhead, production-safe observability/networking/security that previously required custom kernel patches. The verifier and safety model. The ecosystem: bcc, **bpftrace** (the awk-of-the-kernel — most accessible entry point), libbpf, CO-RE (compile once, run everywhere). Real-world uses: deep performance tracing (latency histograms, syscall counts per process), networking (Cilium, XDP fast-path/load-balancing/DDoS drop), security observability (Falco, Tetragon). How it relates to and improves on the `/proc`-based tools from Module 8 and the syscall tracing from Module 7. Honest scope: this is an *intro* — know what it is, why it matters, and run a few canned tools; deep eBPF programming is a specialty.
- **Why it matters / connections:** eBPF is one of the most significant Linux developments of the last decade and is now central to cloud-native observability, networking, and security — strongly resume-relevant. It builds directly on syscalls (Module 7), /proc (Module 8), kernel modules (Module 9, as the thing eBPF lets you *avoid*), and container networking (Module 14/16).
- **Hands-on / exercises:** Install `bpftrace` and run one-liners: count syscalls by process (`bpftrace -e 'tracepoint:raw_syscalls:sys_enter { @[comm] = count(); }'`), trace files opened system-wide, build a latency histogram of a syscall. Run a bcc tool like `execsnoop`/`opensnoop`/`biolatency` and interpret the output. Read one short bpftrace script and explain what it attaches to.
- **You've got it when:** You can explain eBPF as "safe sandboxed programs in the kernel without a module or reboot," and you've used bpftrace/bcc to answer a real question about a running system.

### Module 18: Embedded Linux — Yocto/Buildroot, device tree, BSPs, booting an SBC (major module)
- **Concepts:** What makes embedded Linux different from desktop/server: constrained resources, cross-compilation (Module 13), custom hardware, read-only/robust rootfs, long support lifecycles, no package manager on target (usually). The embedded boot chain on ARM: SoC boot ROM → SPL → **U-Boot** (environment, `bootcmd`, loading kernel+DTB+initramfs, `fastboot`/`tftp`/`mmc` loading) → kernel → init — contrast with the x86/UEFI chain from Module 5. The Linux **device tree** (DT): why it exists (describe non-discoverable hardware to the kernel without hardcoding), `.dts`/`.dtsi` source, the device-tree compiler (`dtc`), nodes/properties/`compatible` strings, how drivers bind to DT nodes (ties to Module 9's platform devices), device-tree overlays. **Board Support Package (BSP):** the vendor-supplied bootloader + kernel + drivers + DT for a specific board. Root filesystem construction approaches and the two dominant build systems: **Buildroot** (simpler, Makefile/Kconfig-based, great for smaller/fixed images — best starting point) vs **Yocto/OpenEmbedded** (industry standard for products: layers, recipes, BitBake, `bblayers`/`local.conf`, machine configs, SDK generation, licensing/SBOM — heavier but production-grade). Embedded filesystems (squashfs read-only rootfs, UBIFS/JFFS2 for raw flash, overlay for writable layers). Init choices on embedded (BusyBox init, systemd, or none). Kernel configuration (`make menuconfig`, defconfigs) and building a kernel + modules + DTB. Flashing/booting: SD card images, `dd`, network boot (TFTP/NFS root for development). The single-board computers: **Raspberry Pi** (most accessible) and **BeagleBone Black** (more "open hardware" with PRUs — interesting given your radar/real-time background).
- **Why it matters / connections:** This is the keystone that fuses everything: cross-compilation (13), boot chain (5), drivers/udev/modules (9), storage/filesystems (15), and your EE/radar hardware instincts. Embedded Linux is a high-demand, defense-relevant, retirement-proof skill area and the natural bridge from topic 04 (embedded) into running real systems. Custom hardware bring-up is exactly the kind of work where your background is differentiating.
- **Hands-on / exercises:** **(Primary capstone-grade task)** Build a custom Linux image with **Buildroot** for a Raspberry Pi (or BeagleBone), flash it to an SD card, and boot it — get a serial console login over UART (your radar-tech soldering/serial skills apply). Then: add a custom package/program to the image. Inspect and modify the **device tree** — enable a peripheral (e.g., I2C/SPI/a GPIO LED) via a DT overlay and drive it from userspace (`/sys/class/gpio` or libgpiod). Interrupt U-Boot at the prompt, inspect/modify its environment, and manually `boot`. For depth, repeat the rootfs build with **Yocto** (a `core-image-minimal` for your board) and compare the experience and output with Buildroot. Cross-compile and `insmod` your Module-9 kernel module *on the SBC*.
- **You've got it when:** You have booted your *own* Linux image on physical SBC hardware to a login prompt, modified its device tree to enable and drive a peripheral, and can explain the full ARM boot chain (boot ROM → SPL → U-Boot → kernel + DTB → init) and the trade-off between Buildroot and Yocto.

### Module 19: Infrastructure tooling & putting it in production (MODERN, intro)
- **Concepts:** How modern teams manage Linux fleets at scale instead of by hand. Configuration management / infrastructure-as-code: Ansible (agentless, SSH-based, playbooks/inventory/roles — the most accessible and widely demanded; MODERN) with a pointer to Terraform (provisioning cloud infra) and the declarative philosophy. Remote access at scale: SSH config, jump hosts/bastions, key management. Secrets management (intro: don't put secrets in git; vaults). Time/scheduling: cron (classic) vs systemd timers (Module 6). Logging/metrics/observability stack at a high level (journald → log aggregation; Prometheus/Grafana for metrics; ties to eBPF, Module 17). Immutable infrastructure and the container/orchestration horizon (Module 14 → Kubernetes as the scale-out endpoint). CI/CD as it touches Linux (build/test in containers — cross-ref topic 10). Honest scope: this is orientation so the *vocabulary* and *landscape* are familiar in interviews and on the job, not mastery of each tool.
- **Why it matters / connections:** "Can you configure one Linux box" is table stakes; "can you manage a fleet reproducibly" is the modern job. Ansible/IaC fluency is broadly requested. This module connects the whole topic outward to topic 10 (engineering practice) and to real employment.
- **Hands-on / exercises:** Write a small Ansible playbook that installs a package, drops a config file from a template, and ensures the systemd service from Module 6 is enabled and running — run it against a VM/container. Set up passwordless SSH and a hardened `~/.ssh/config` with a jump host entry. Convert one of your earlier manual setups (e.g., the Module 15 storage or Module 16 firewall config) into an idempotent playbook.
- **You've got it when:** You can express "set this machine up exactly like this" as an idempotent, re-runnable artifact rather than a list of remembered commands.

## Capstone / integrative exercise

**Build, harden, observe, and ship a self-contained Linux appliance — then put the same workload on custom embedded hardware.**

Part A — Server/container track:
1. On a fresh VM, provision storage with **LVM** (Module 15) for the service's data directory, mounted via UUID in `/etc/fstab`.
2. Write a small network service (in your language of choice) and **containerize it** with a multi-stage Dockerfile running as a non-root user; run it under **rootless Podman** (Module 14).
3. Manage it with a **hardened systemd unit** (Quadlet/`systemctl --user` or system service) with `Restart=on-failure`, resource limits, and security directives; add a **systemd timer** for a nightly backup script (Modules 6, 12).
4. Lock down the host with an **nftables** ruleset allowing only SSH and the service port; configure the interface with `ip`/networkd (Module 16).
5. **Observe** it: read its logs via `journalctl`, watch resource use via `/proc` and cgroups, and use a **bpftrace/bcc** one-liner to confirm what syscalls/files it touches under load (Modules 7, 8, 17).
6. Express the entire host setup as an **idempotent Ansible playbook** (Module 19).

Part B — Embedded track:
7. **Cross-compile** the same service for ARM (Module 13).
8. Build a custom rootfs with **Buildroot** that includes it, **boot it on a Raspberry Pi/BeagleBone** to a serial console, and run the service on the SBC (Module 18).
9. Modify the **device tree** to enable a GPIO/peripheral and have the service (or a helper) drive it; load your own **kernel module** (Module 9) on the target.

Deliverable: a short write-up tracing one user request from the wire, through nftables, the kernel network stack, a syscall into your containerized process, to the LVM-backed disk — naming the subsystem at each hop. This single thread exercises essentially every module.

## Common pitfalls & rust-knockers

- **Quoting/word-splitting bugs:** unquoted `$var`, `for f in $(ls)`, forgetting `read -r`. The classic that bites everyone. (Run ShellCheck.)
- **Legacy-tool muscle memory:** reaching for `ifconfig`/`route`/`netstat`/`iptables` when `ip`/`ss`/`nft` are the current tools. Retrain this deliberately — it dates you in interviews. (MODERN)
- **`sudo` and PATH/environment surprises:** `sudo` resets the environment; `sudo command` vs `sudo -i` differ; a script that works for you fails under a service account with a minimal env.
- **systemd `Type=` mismatch:** declaring `Type=simple` for a forking daemon (or vice versa) so systemd misreads readiness/state. Know simple vs forking vs notify vs oneshot.
- **Permissions vs capabilities confusion:** slapping setuid root on things instead of granting a specific capability; not understanding that `x` on a directory means "traverse."
- **`SIGKILL`/`D`-state expectations:** thinking `kill -9` always works (it can't reap a D-state process or flush its work).
- **Editing distro-shipped unit files in `/usr/lib`** instead of using `/etc/systemd/system` drop-ins (`systemctl edit`) — your changes get overwritten on update.
- **Forgetting `daemon-reload`** after editing units, or `enable` vs `start` confusion (enable = at boot, start = now).
- **Container ≠ VM confusion:** expecting kernel isolation, running everything as root in the image, baking secrets into layers, ignoring that the container shares the host kernel.
- **Cross-compilation foot-guns:** accidentally linking host libraries, glibc-vs-musl ABI mismatches, wrong sysroot, confusing build/host/target.
- **Device tree mistakes:** wrong `compatible` string so the driver never binds; editing the DTB instead of the `.dts`; forgetting to recompile/redeploy the DTB.
- **fstab errors that brick boot:** a bad `/etc/fstab` entry can drop you to emergency mode — always test mounts before relying on them; know how to recover from the boot module.
- **`rm -rf` / `dd` with a wrong path or device** — destructive and irreversible; double-check `lsblk` before `dd of=/dev/...`.
- **Assuming Linux from one distro is universal:** package managers, init details, and network config tooling differ across Debian/RHEL/Arch families.
